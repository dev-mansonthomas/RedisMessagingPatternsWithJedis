🧠 Pseudo-code / logic

🧩 Redis features used on this page

  • Streams — the conversation (chat:cid), per-response tokens (chat:cid:tok), moderation flags and the DLQ are all streams (XADD, XRANGE/XREVRANGE, XLEN, XREAD).
  • Consumer groups — fan-out — three groups read the same stream independently: cg:responder, cg:moderation, cg:analytics (XREADGROUP, XACK, XINFO GROUPS, XPENDING).
  • Crash recovery + DLQ — reclaim stale-pending messages and give up after N tries (XAUTOCLAIM → chat:cid:dlq).
  • Approximate trimming — bound stream memory with MAXLEN ~.
  • Hashes — analytics counters in chat:cid:stats (HINCRBY, HGETALL).
  • RedisTimeSeries — the tokens-over-time chart ts:cid:userTokens (TS.ADD, TS.RANGE … AGGREGATION).
  • Key expiry / TTL — transient token stream and the reply-timeout key (SETEX, EXPIRE).
  • Keyspace notifications — reply timeout driven by key-expiry events (notify-keyspace-events Ex, PSUBSCRIBE __keyevent@0__:expired).

👤 UI → POST /message(text)
function  postMessage(cid, text):
ensureConversation(cid)
// XGROUP CREATE (idempotent)
XADD chat:cid role=user
// group before XADD → no miss
2 responder generates (1 VT/cid)
loop:
id,msg = XREADGROUP cg:responder
if role != user: XACK; skip
ctx = XREVRANGE chat:cid ..id
llm.generate(ctx):
onToken: XADD chat:cid:tok
onComplete: XADD assistant
EXPIRE tok 60; XACK
3 tokens stream to the user
loop:
es = XREAD chat:cid:tok lastId
for e in es:
ws.push(cid, TOKEN, e.token)
👤 UI renders token
1

The user posts a message. The browser POSTs it; ensureConversation creates the stream and the cg:responder group if absent (idempotent, MKSTREAM) and lazily starts the per-conversation workers, then XADD chat:cid appends the turn — the conversation is the stream (ordered, replayable).

2

The responder generates the reply. One Virtual Thread blocks on XREADGROUP cg:responder (it wakes on a new message — no polling), skips non-user turns, rebuilds the context with XREVRANGE, calls the LLM, XADDs each generated token to chat:cid:tok, then appends the finished assistant turn and XACKs the user message.

3

Tokens stream back to the user. A second Virtual Thread XREADs the token stream and pushes each token over WebSocket (only to this cid's subscribers); the UI appends them, so the answer appears word by word, at the pace the LLM generates it.

🔀 Fan-out: the same chat:{{ cid }} stream also feeds two more consumer groups in parallel — cg:moderation (flags user messages containing a keyword → chat:{{ cid }}:flags) and cg:analytics (counters + TS.ADD to ts:{{ cid }}:userTokens). Three groups, one stream, no copy — see the internals panel above.

♻️ Resilience: a per-cid sweeper XAUTOCLAIMs messages a worker took but never XACKed (idle beyond minIdle) and regenerates them — so a crash mid-generation still gets a reply (Kill & recovery). After maxDeliveries failed attempts the message is routed to chat:{{ cid }}:dlq (Kill & DLQ).

⏱ Reply timeout (Redis keyspace notifications): each message sets a short-lived key llm:timeout:{{ '{' }}msgId{{ '}' }} that's deleted when the reply completes. If it expires first, Redis fires a keyspace expired event (notify-keyspace-events Ex) that a listener turns into a ⏱ system notice here — so you're told when a message failed to get a reply, without polling. Same mechanism as Request/Reply (ADR-0007).
